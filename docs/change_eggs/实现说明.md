# 换蛋广场 + 用户私聊/通知 实现说明

> 本文档对应需求 `换蛋匹配.md`，记录两块功能的实现逻辑、表结构、接口与改动文件，便于联调与后续维护。

## 一、功能概览

1. **换蛋广场**：用户上传"我拥有的蛋组"与"我需求的蛋组"（均记录 `pokemon_id` + `tag`），附带游戏 id。系统通过**定时任务**周期扫描 + **提交时立即异步触发**两种方式，找出"需求与被需求双向吻合"的两个用户，命中后给双方下发通知。
2. **用户私聊 / 通用消息下发**：A 给 B 发消息先落库；B 在线则通过 WebSocket 实时下发，离线则等 B 上线后补发。封装为统一入口 `send_message`，换蛋匹配通知直接复用它。

### 已确认的关键决策
- **匹配条件**：双向 `pokemon_id + tag` 都必须一致 —— `A.拥有 == B.需求` 且 `A.需求 == B.拥有`（pokemon_id 与 tag 同时相等）。
- **发送入口**：扩展 `/ws/{user_id}` 接收发送帧（实时）+ 新增 REST `POST /api/messages`（兜底/离线落库）。
- **定时任务**：引入 **APScheduler**（`AsyncIOScheduler` interval 任务）；提交挂单时另用 `asyncio.create_task` 立即触发一次匹配。

代码遵循现有三层结构 `routes → services → repositories` + `schemas`；面向 C 端接口沿用 `album` 约定（`user_id` 直接走请求体 / Query，不挂鉴权）。

## 二、数据库表

建表语句见 `sql/change_egg.sql`，并在仓储 `ensure_*()` 内用 `CREATE TABLE IF NOT EXISTS` 在应用启动时自动建立，**无需手动执行**。

### 1. `change_egg_listing` — 换蛋挂单
| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | BIGSERIAL PK | |
| `user_id` | BIGINT | 平台用户 id |
| `game_id` | VARCHAR(64) | 游戏 id（长串数字） |
| `own_pokemon_id` | INT | 拥有的精灵 id（`pokemon.id`） |
| `own_tag` | VARCHAR(30) | 拥有蛋组 tag code |
| `want_pokemon_id` | INT | 需求的精灵 id（`pokemon.id`） |
| `want_tag` | VARCHAR(30) | 需求蛋组 tag code |
| `status` | VARCHAR(16) | `open` / `matched` / `closed` |
| `created_at` / `updated_at` | TIMESTAMP | |

索引：`idx_cel_want(want_pokemon_id, want_tag, status)`、`idx_cel_own(own_pokemon_id, own_tag, status)`、`idx_cel_user(user_id, status)`。

> 无论用户从 `pokemon_egg` 蛋组还是 `pokemon` 列表选取，最终都只存 `pokemon.id`。

### 2. `change_egg_match` — 命中记录（去重）
| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | BIGSERIAL PK | |
| `listing_a_id` / `listing_b_id` | BIGINT | 命中的两条挂单（写入时归一化保证 `a < b`） |
| `user_a_id` / `user_b_id` | BIGINT | 对应双方用户 |
| `created_at` | TIMESTAMP | |

唯一约束 `uk_cem_pair(listing_a_id, listing_b_id)` —— 同一对挂单只命中一次，避免重复通知（定时任务全量扫描时靠它去重）。

### 3. `user_message` — 私聊 / 通用消息
| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | BIGSERIAL PK | |
| `from_user_id` | BIGINT | 发送方（**系统通知 = 0**） |
| `to_user_id` | BIGINT | 接收方 |
| `msg_type` | VARCHAR(20) | `chat` / `egg_match_notify` |
| `content` | TEXT | 文本内容 |
| `payload` | JSONB | 结构化附加数据（可空，如匹配双方信息） |
| `is_delivered` | BOOLEAN | 是否已 ws 下发 |
| `delivered_at` | TIMESTAMP | |
| `is_read` | BOOLEAN | 是否已读 |
| `created_at` | TIMESTAMP | |

索引：`idx_um_pending(to_user_id, is_delivered)`（拉未投递）、`idx_um_conv(from_user_id, to_user_id, id)`（拉会话）。

### tag 字典（复用 `sys_dict`，不新建表）
tag 存入 `sys_dict`，`dict_type='egg_exchange_tag'`，`code`/`label` = 大块头、小不点等。
- 后台维护：复用现有 `GET/POST /api/ops/dicts?dict_type=egg_exchange_tag`。
- 种子数据：`sql/change_egg.sql` 末尾已附 `INSERT ... ON CONFLICT DO NOTHING`。

## 三、换蛋匹配逻辑

### 匹配查询（`change_egg_repository.find_reciprocal_matches`）
```sql
SELECT ... FROM change_egg_listing
WHERE status = 'open'
  AND user_id <> :user_id                              -- 排除自己
  AND own_pokemon_id = :want_pokemon_id AND own_tag = :want_tag   -- 对方拥有 == 我需求
  AND want_pokemon_id = :own_pokemon_id AND want_tag = :own_tag   -- 对方需求 == 我拥有
```

### 编排（`change_egg_service`）
- **`run_match_for_listing(listing_id)`**：取挂单 → `find_reciprocal_matches` → 对候选 `record_match`（唯一约束去重，已存在则跳过）→ 双方 `mark_listings_matched` 置 `matched` → 调 `message_service.send_message(msg_type='egg_match_notify')` 给双方各发一条通知 → 命中一对即停止（当前挂单已匹配）。
- **`run_match_all_open()`**：扫描所有 `open` 挂单逐个跑 `run_match_for_listing`，由定时任务调用，靠 `change_egg_match` 去重保证不重复通知。

### 触发时机
- **提交挂单后**：`api/routes/change_eggs.py` 在创建后 `asyncio.create_task(run_match_for_listing(id))`，**不阻塞响应**。
- **定时**：`change_egg_scheduler.py` 用 APScheduler 每 **2 分钟**（环境变量 `CHANGE_EGG_MATCH_INTERVAL_MINUTES` 可覆盖）跑一次 `run_match_all_open`。

### 匹配通知内容
`payload` 结构（`kind="egg_match"`）含对方 `user_id`、`game_id`、挂单 id 与拥有/需求的 pokemon_id+tag，前端可据此引导用户加好友/换蛋。

## 四、通用私聊 / 消息下发

### 统一入口 `message_service.send_message(from_user_id, to_user_id, content, msg_type='chat', payload=None)`
1. `insert_message` 落库；
2. 接收方在线（`manager.active_connections` 含 `str(to_user_id)`）→ `manager.send_json` 实时下发 + `mark_delivered`；
3. 返回消息记录。

### `message_service.flush_undelivered(user_id)`
用户上线时调用：查 `list_undelivered` → 逐条 ws 下发 → `mark_delivered`，返回补发条数。

### ws 事件结构（前端按 `node` 路由）
```json
{
  "node": "message",
  "type": "chat | egg_match_notify",
  "id": 123,
  "from_user_id": 1001,
  "to_user_id": 1002,
  "content": "...",
  "payload": { ... } ,
  "created_at": "2026-06-11T12:00:00"
}
```

### 在线判断
`/ws/{user_id}` 以 `str(user_id)` 为键注册连接，故 `str(to_user_id) in manager.active_connections` 即在线。

## 五、接口一览

### 换蛋广场（prefix `/api/change-eggs`）
| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `` | 创建挂单，落库后**立即异步匹配**，即时返回挂单 |
| GET | `` | 查我的挂单（Query `user_id`，可选 `status` 筛选：open/matched/closed） |
| GET | `/square` | 广场分页浏览所有 `open` 挂单（可选 `pokemon_id` / `tag` 过滤） |
| GET | `/tags` | 公开返回可选蛋组 tag |
| POST | `/{listing_id}/close` | 用户主动关闭自己正在匹配的挂单：status `open → closed`（Body `user_id` 校验归属） |
| DELETE | `/{listing_id}` | 彻底删除自己的挂单（任意状态，Query `user_id` 校验归属） |

创建挂单请求体：
```json
{
  "user_id": 1001,
  "game_id": "1234567890",
  "own_pokemon_id": 10,
  "own_tag": "大块头",
  "want_pokemon_id": 20,
  "want_tag": "小不点"
}
```

### 私聊消息（prefix `/api/messages`）
| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `` | 发送消息 `{from_user_id, to_user_id, content}`，走统一入口 |
| GET | `/inbox` | 收件箱：我收到的消息/通知（Query `user_id, limit`，按时间倒序） |
| GET | `/unread-count` | 未读消息数（Query `user_id`），用于小程序红点提醒 |
| GET | `/conversation` | 拉会话历史（Query `user_a, user_b, limit`，双向、按时间正序） |
| POST | `/read` | 标记已读 `{to_user_id, ids: []}` |

### WebSocket（`/ws/{user_id}`，已扩展）
- 连接建立 → 自动 `flush_undelivered` 补发离线消息；
- 用户发送私聊帧：
```json
{ "type": "chat", "to": 1002, "content": "你好" }
```
  → 转交 `handle_user_ws_message` → `send_message` 落库 + 实时下发给接收方。
- 其它类型帧忽略；原有"接收后端流式事件"能力保留。

## 七、验证步骤

1. `uv sync` 拉入 apscheduler；`uv run python -m compileall api ws` 语法检查（已通过）。
2. `uv run uvicorn api.main:app --reload --port 8000`，Swagger `/docs` 检查新接口；启动日志确认调度器已启动、三张表已建。
3. **换蛋匹配**：用两个 user_id 互为镜像创建挂单（A 拥有X需求Y、B 拥有Y需求X，tag 一致）→ 第二次提交应在数秒内产生 `change_egg_match` 记录 + 两条 `egg_match_notify` 消息。tag 改成不一致复测应**不**匹配。
4. **私聊在线**：两个 `wscat` 连 `/ws/{A}`、`/ws/{B}`；A 发 `{"type":"chat","to":B,"content":"hi"}`，B 端实时收到 `node:message` 事件，库中 `is_delivered=true`。
5. **离线补发**：仅连 A，`POST /api/messages` 发给离线的 B → 入库 `is_delivered=false`；再连 `/ws/{B}`，连接瞬间收到补发并置 `is_delivered=true`。
6. **REST 历史**：`GET /api/messages/conversation?user_a=A&user_b=B` 返回上述消息。
