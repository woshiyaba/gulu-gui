# 运营维护平台（`/ops`）设计文档

## 1. 背景与目标

`ops` 平台定位为「核心数据维护」阶段，重点是维护效率与可用性。  
当前里程碑已覆盖：登录、字典、用户、精灵（含技能关系、进化链、立绘上传）、技能库（含图标上传、占用查询、强制删除）。

本文目标：

1. 记录当前已实现能力（与代码一致）
2. 沉淀精灵立绘上传、技能维护的交互/接口契约
3. 标注已知问题与下一步迭代方向

## 3. 设计原则

- 接口前缀统一：`/api/ops/*`
- 分层一致：`routes -> services -> repositories`
- 严格权限：`editor` 可新增/修改/上传，`admin` 可删除/强制删除
- 写操作可追踪：写入 `ops_audit_log`
- 事务优先：主表与子表变更保持原子性
- 与查询链路解耦：不改 `/api/pokemon*` 对外契约
- 图片上传统一走 `_save_image_upload`，不同资源只是落盘目录与外链前缀不同

## 4. 数据模型与维护范围

核心涉及表：

- `pokemon`
- `pokemon_attribute`
- `pokemon_egg_group`
- `pokemon_skill`（pokemon_id × skill_id，带 `type`、`sort_order`）
- `pokemon_trait`
- `attribute`
- `skill`（技能库本体）
- `evolution_chain`

字段策略：

- `pokemon.type/type_name`、`form/form_name`、`egg_group` 为业务冗余直存
- `pokemon_skill.type` 业务值直存（`原生技能/学习技能`）
- `pokemon.image_lc`：朋友图文件名；对外 URL = `FRIEND_IMAGE_BASE_URL` + 文件名
- `skill.attr_id`：外键到 `attribute.id`；对外/前端统一以「属性名」（字符串）交换，仓储层在读写时做 `name ↔ id` 映射
- `skill.icon`：相对路径或文件名
  - 兼容老数据（包含 `/` 的路径）时走 `build_image_url`（`BASE_URL=rocom.game-walkthrough.com`，并自动剥掉下划线编号前缀）
  - 新上传（仅文件名）时走 `SKILL_ICON_BASE_URL`（默认 `https://wikiroco.com/images/icon/skill/`）

## 5. 精灵维护

### 5.1 列表筛选与分页

- 后端分页固定：`page/page_size`
- 当前筛选参数：
  - `no`（精确）
  - `name`（like）
  - `type_code`
  - `form_code`
  - `trait_id`
- 特性筛选使用「输入联想 + 选项提示」，最终提交 `trait_id` 精确过滤

### 5.2 编辑弹窗

1. 基础信息 + 立绘（左右双栏，移动端自动堆叠）
   - 阶段/形态编码下拉；阶段/形态名称自动联动只读
   - 立绘区域：点击预览或「更换/上传」选图；仅做本地 blob 预览；「移除」清空
2. 种族值
   - 子项可改，总和自动计算只读
3. 关联信息
   - 属性 pill（最多 2 种）
   - 蛋组 pill 多选
4. 技能列表
   - 名称搜索、图标展示、分页
   - 新增技能弹窗检索并单选加入（候选来自 `/api/ops/pokemon/options` 的 `skills`）
5. 进化链（可视化）
   - 链卡片流（图片 + 名称 + 条件）
   - 按精灵名搜索已有链
   - 当前精灵卡片可拖拽到链中位置插入

### 5.3 保存策略（含立绘）

点击底部「保存」时按顺序执行：

1. 若存在待上传文件，先调 `POST /api/ops/pokemon/friend-image`，把返回的 `image_lc` 回填表单；失败则中止
2. `POST/PUT /api/ops/pokemon*` 保存主数据（含 `image_lc`）
3. 若进化链有有效步骤，调 `PUT /api/ops/pokemon/{id}/evolution-chain`

### 5.4 立绘上传契约

- 接口：`POST /api/ops/pokemon/friend-image`
- 鉴权：`editor / admin`
- 请求：`multipart/form-data`，字段 `file`
- 响应：`{ image_lc, preview_url }`
- 限制：`.webp/.png/.jpg/.jpeg/.gif`、≤ 5MB、非空
- 落盘：`FRIEND_IMAGE_UPLOAD_DIR`（默认 `/var/www/images/friends`，可用环境变量覆盖）
- 冲突：同名自动加 `_2/_3 ...` 后缀；无合法 stem 时用 UUID
- 对外 URL：`FRIEND_IMAGE_BASE_URL` + `image_lc`，需 Nginx 把 `/images/friends/` 映射到同一磁盘目录

## 6. 技能维护

目标：把当前只能在精灵编辑里「选技能」的只读候选，升级为完整的技能库维护入口，已覆盖新增/修改/删除、图标上传、占用查询、强制删除。

### 6.1 页面与路由

- 路由：`/ops/skills`
- 列表 + 弹窗式编辑（与精灵维护风格一致）
- 列表列：序号、图标、名称、属性、类型（物攻/魔攻/状态/防御）、威力、消耗、描述简写、操作

### 6.2 列表与筛选

- 请求：`GET /api/ops/skills`
- 参数：`keyword`（名称 like）、`attr`（属性名）、`type`（技能类型）、`page`、`page_size`
- 排序：`ORDER BY s.id DESC`
- 分页：后端分页

### 6.3 编辑弹窗（字段）

一一对应 `skill` 表：

| 字段 | UI | 说明 |
|---|---|---|
| `name` | 必填文本 | 技能名称；`UNIQUE`；长度 ≤ 50；服务端做重名校验 |
| `attr` | 下拉 | 属性名；选项来自 `attribute.name`；空值表示不限定属性 |
| `type` | 下拉 | 物攻 / 魔攻 / 状态 / 防御；选项优先取 `sys_dict('skill_type')`，没维护时回退到现有 `skill.type` 去重 |
| `power` | 数字 | 威力；`0 ~ 9999` |
| `consume` | 数字 | 消耗；`0 ~ 9999` |
| `skill_desc` | 多行文本 | 技能描述 |
| `icon` | 图片上传 + 预览 | 交互与精灵立绘完全一致 |

保存时：

- 重名 → 400 `技能名称已存在`
- 威力/消耗越界 → 400
- `attr` 传入属性名，后端查 `attribute.id` 写入 `attr_id`；查不到则存 `NULL`

### 6.4 图标上传

- 接口：`POST /api/ops/skills/icon`（multipart/form-data；字段 `file`）
- 鉴权：`editor / admin`
- 限制：`.webp/.png/.jpg/.jpeg/.gif`、≤ 2MB
- 落盘：`SKILL_ICON_UPLOAD_DIR`，默认 `/var/www/images/icon/skill`（可用环境变量覆盖）
- 响应：`{ icon, preview_url }`；`icon` 是落盘文件名
- 冲突：同名自动加 `_2/_3 ...`；无合法 stem 时用 UUID
- 交互：与立绘一致，只做本地预览，点「保存」时再一并上传并写库
- 需要 Nginx 将 `/images/icon/skill/` 映射到同一磁盘目录

### 6.5 删除与占用

- 默认删除：`DELETE /api/ops/skills/{id}`
  - 服务端先查 `pokemon_skill` 占用，有占用 → 409 Conflict，附带占用数量
- 查占用：`GET /api/ops/skills/{id}/usages`
  - 返回占用精灵列表（精灵编号、名称、来源 `原生/学习技能`、排序）
  - 前端在列表行「占用」按钮打开弹窗展示
- 强制删除：`DELETE /api/ops/skills/{id}?force=1`
  - 仅 `admin` 可用；先 `DELETE FROM pokemon_skill WHERE skill_id = %s` 再删技能
  - 审计日志 `action = force_delete`，记录 before
- 前端 409 时：`admin` 可在弹窗二次确认后直接发起 `force=1`；`editor` 会收到提示 Toast

### 6.6 接口清单

- `GET /api/ops/skills`（列表，分页）
- `GET /api/ops/skills/options`（`attrs`, `types`）
- `GET /api/ops/skills/{id}`（详情）
- `POST /api/ops/skills`（新增）
- `PUT /api/ops/skills/{id}`（修改）
- `DELETE /api/ops/skills/{id}?force=0|1`
- `GET /api/ops/skills/{id}/usages`（查占用）
- `POST /api/ops/skills/icon`（上传图标）

### 6.7 鉴权

- `editor`：读、新增、修改、上传图标、普通删除（前提是无占用）
- `admin`：额外可强制删除（级联清 `pokemon_skill`）

### 6.8 数据兼容策略

- `pokemon_skill` 仍走 `pokemon_id × skill_id`，技能改名不影响关系
- `skill.attr` 字段在 PG schema 中实际为 `attr_id` 外键；对外 API 统一为属性名，读写在仓储层做映射
- 旧导入脚本（`scripts/import_skill_stones.py` / `scripts/migrate_mysql_to_pg*.py`）保持不变；`ops/skills` 仅作为维护入口，不接管导入
- 老 `skill.icon` 中带路径前缀的数据继续由 `build_image_url` 负责拼 `BASE_URL`；新上传统一纯文件名 + `SKILL_ICON_BASE_URL`

### 6.9 UI 要点

- 列表右上角「新增」按钮
- 列表每行「修改 / 占用 / 删除」三连
- 编辑弹窗两列网格：名称、属性、类型、威力、消耗 + 整行描述 + 整行图标
- 图标预览 72×72 方形虚线框；未上传显示「无图标」
- 描述列过长自动省略，鼠标悬停 title 显示完整
- 错误样式：与全局 ops 一致（红底白字 + Toast）

## 7. 权限设计

- `editor`
  - 精灵：列表/详情/新增/修改（含立绘上传、进化链维护）
  - 技能：列表/详情/新增/修改、图标上传、普通删除（无占用）
- `admin`
  - 全部操作（含精灵删除、技能强制删除）

通用规则：

- 所有 `/api/ops/*` 都要求登录态
- 删除类操作必须二次确认；破坏性操作（含 `force`）需要显式确认并写入审计日志
