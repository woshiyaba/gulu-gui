# 精灵阵容系统设计

> 目标：在当前项目已有的精灵图鉴、技能库、性格字典、血脉字典、Ops 后台能力基础上，设计一套可维护的“精灵阵容”系统。
>
> 该系统用于配置一套最多 6 只精灵的推荐阵容。每只精灵可配置：精灵本体、血脉、性格、三项资质、四个技能，以及单精灵说明；整个阵容可配置整体说明。

---

## 1. 设计背景

### 1.1 当前系统已有能力

结合当前仓库，已经具备以下基础能力：

- 精灵基础资料已存在于 `pokemon` 表；
- 技能资料已存在于 `skill` 表；
- 精灵性格字典已存在于 `personality` 表，并已有公共接口与 Ops 维护能力；
- 精灵血脉字典已通过 `sys_dict` 管理，`dict_type = 'pet_bloodline'`；
- 已有一个“最多 6 只精灵 + 每只 4 个技能 + 整体说明”的落地模块：`starlight_duel_episode` / `starlight_duel_pet`。

因此，新的“精灵阵容系统”不建议重新发明一套完全不同的结构，而应当：

- 复用 `pokemon` / `skill` / `personality` / `sys_dict`；
- 延续 `starlight_duel` 的后台维护体验；
- 在其基础上补齐：血脉、性格、三项资质、单精灵备注、阵容备注。

### 1.2 核心业务目标

支持运营或攻略作者配置一套推荐阵容，并在前台给用户展示：

- 阵容名称；
- 阵容说明；
- 阵容中的 1~6 只精灵；
- 每只精灵的完整搭配信息；
- 每只精灵的补充说明。

---

## 2. 需求边界

### 2.1 阵容层

每套阵容包含：

- 阵容名称；
- 阵容说明；
- 状态（启用 / 禁用）；
- 排序值；
- 最多 6 只精灵。

### 2.2 单精灵层

每只精灵配置项包括：

- 精灵：关联 `pokemon.id`；
- 血脉：关联 `sys_dict` 中 `dict_type='pet_bloodline'`；
- 性格：关联 `personality.id`；
- 三项资质；
- 四个技能：关联 `skill.id`；
- 单精灵说明：可为空；
- 阵容内顺序：1~6。

### 2.3 三项资质的定义

“三项资质”在本系统中不建议写死为固定的三种属性，而应设计为：

- 一只精灵固定有 3 个资质槽位；
- 每个槽位包含：
  - 一个种族值字段标识；
  - 一个资质值；
- 种族值字段从以下 6 个中选择：
  - `hp`
  - `atk`
  - `matk`
  - `def_val`
  - `mdef`
  - `spd`

这样可以覆盖不同精灵的培养方案，例如：

- 物攻流：`atk / spd / hp`
- 魔攻流：`matk / spd / hp`
- 坦克流：`hp / def_val / mdef`

### 2.4 可为空规则

- 阵容说明：可为空；
- 单精灵说明：可为空；
- 血脉：建议可为空；
- 性格：建议可为空；
- 技能 1~4：可少于 4 个，空位表示“可自由配置”；
- 阵容精灵数量：可少于 6 只。

### 2.5 本期不做

- 不做玩家个人配队保存；
- 不做阵容点赞、收藏、评论；
- 不做复杂筛选推荐算法；
- 不做阵容复制/版本历史；
- 不做战斗模拟计算。

## 3. 数据建模方案

基于当前项目风格，建议采用“主表 + 子表”的结构，并在子表中继续使用固定槽位字段，保持和 `starlight_duel_pet` 风格一致，降低开发成本。

### 3.1 主表：阵容表

```sql
CREATE TABLE IF NOT EXISTS pokemon_lineup (
    id            SERIAL       PRIMARY KEY,
    title         VARCHAR(100) NOT NULL DEFAULT '',
    lineup_desc   TEXT         NOT NULL DEFAULT '',
    source_type   VARCHAR(30)  NOT NULL DEFAULT '', -- 可预留：event/pve/pvp/boss/manual
    sort_order    INT          NOT NULL DEFAULT 0,
    is_active     BOOLEAN      NOT NULL DEFAULT TRUE,
    created_at    TIMESTAMP    NOT NULL DEFAULT NOW(),
    updated_at    TIMESTAMP    NOT NULL DEFAULT NOW()
);
```

字段说明：

- `title`：阵容标题，例如“系别赛通用水系阵容”；
- `lineup_desc`：整套阵容说明；
- `source_type`：预留分类字段，便于未来区分活动/BOSS/PVP；
- `sort_order`：运营排序；
- `is_active`：是否展示。

### 3.2 子表：阵容精灵表

```sql
CREATE TABLE IF NOT EXISTS pokemon_lineup_member (
    id               SERIAL PRIMARY KEY,
    lineup_id        INT    NOT NULL REFERENCES pokemon_lineup(id) ON DELETE CASCADE,
    sort_order       INT    NOT NULL DEFAULT 1,

    pokemon_id       INT    NOT NULL REFERENCES pokemon(id),
    bloodline_dict_id INT   REFERENCES sys_dict(id),
    personality_id   SMALLINT REFERENCES personality(id),

    qual_1_stat_key  VARCHAR(20) NOT NULL DEFAULT '',
    qual_1_value     INT         NOT NULL DEFAULT 0,
    qual_2_stat_key  VARCHAR(20) NOT NULL DEFAULT '',
    qual_2_value     INT         NOT NULL DEFAULT 0,
    qual_3_stat_key  VARCHAR(20) NOT NULL DEFAULT '',
    qual_3_value     INT         NOT NULL DEFAULT 0,

    skill_1_id       INT REFERENCES skill(id),
    skill_2_id       INT REFERENCES skill(id),
    skill_3_id       INT REFERENCES skill(id),
    skill_4_id       INT REFERENCES skill(id),

    member_desc      TEXT NOT NULL DEFAULT '',

    CONSTRAINT uk_lineup_member_order UNIQUE (lineup_id, sort_order)
);
```

### 3.3 为什么资质采用“3 个固定槽位”

虽然三项资质也可以拆成独立子表，但结合当前项目实际，固定槽位更合适：

- 当前 `starlight_duel_pet` 已经使用 `skill_1_id ~ skill_4_id` 的固定列设计；
- 本需求明确只有“三项资质”，上限固定；
- 固定列更利于 Ops 表单开发与接口序列化；
- 当前阶段无需为任意数量资质做高度抽象。

如果未来出现“四项资质”“资质模板复用”等需求，再拆子表也不迟。

## 4. 字段与约束设计

### 4.1 血脉字段

血脉不单独建表，直接复用 `sys_dict`：

- `dict_type = 'pet_bloodline'`
- `code`：业务编码
- `label`：显示名称

保存时需要校验：

- `bloodline_dict_id` 对应字典必须存在；
- 且必须属于 `pet_bloodline` 类型。

### 4.2 性格字段

直接复用 `personality` 表：

- 后台下拉展示 `name_zh`；
- 前台可展示性格名称以及对应的加减项；
- 本阵容系统只保存 `personality_id`，不冗余修正值。

### 4.3 三项资质字段约束

建议在 service 层校验：

- `qual_1_stat_key ~ qual_3_stat_key` 只能从 `hp/atk/matk/def_val/mdef/spd` 中选择；
- 三个 `stat_key` 不允许重复；
- `qual_*_value` 建议为非负整数；
- 三项资质必须成组出现，不允许出现“有 key 无 value”或“有 value 无 key”。

资质值的数据类型建议先用 `INT`，由业务文案决定展示单位，例如：

- 若资质本质是“个体值”，可直接显示整数；
- 若后续需要“31/31”“252”之类更复杂模型，再扩展。

### 4.4 技能字段约束

建议约束：

- 4 个技能槽允许为空；
- 非空技能不允许重复；
- 不强制技能必须属于该精灵可学习技能，原因如下：
  - 当前很多攻略配置会包含“理论推荐技”或“可替换技”；
  - 若强校验，会显著增加运营维护复杂度；
  - 可以先放宽，后续若需要再加“仅允许该精灵已有技能”的可选校验。

### 4.5 数量约束

建议在 service 层限制：

- 每套阵容最少 1 只精灵；
- 最多 6 只精灵；
- `sort_order` 必须连续整理为 `1..N`；
- 保存时按前端传入顺序重排，而不是相信客户端原值。

## 5. 后端分层设计

建议完全对齐当前 `starlight_duel` 的三层结构：

```text
api/
├── schemas/pokemon_lineup.py
├── repositories/pokemon_lineup_repository.py
├── services/pokemon_lineup_service.py
└── routes/
    ├── ops.py
    └── pokemon.py
```

### 5.1 Schema 设计

建议定义以下模型：

```python
class PokemonLineupMemberInput(BaseModel):
    pokemon_id: int
    sort_order: int = 1
    bloodline_dict_id: int | None = None
    personality_id: int | None = None

    qual_1_stat_key: str = ""
    qual_1_value: int = 0
    qual_2_stat_key: str = ""
    qual_2_value: int = 0
    qual_3_stat_key: str = ""
    qual_3_value: int = 0

    skill_1_id: int | None = None
    skill_2_id: int | None = None
    skill_3_id: int | None = None
    skill_4_id: int | None = None

    member_desc: str = ""


class PokemonLineupUpsertRequest(BaseModel):
    title: str = ""
    lineup_desc: str = ""
    source_type: str = ""
    sort_order: int = 0
    is_active: bool = True
    members: list[PokemonLineupMemberInput] = Field(default_factory=list)
```

详情返回模型中建议补充展示字段，减少前端拼装：

- `pokemon_name`
- `pokemon_image`
- `bloodline_label`
- `personality_name_zh`
- `skill_1_name ~ skill_4_name`

### 5.2 Repository 设计

Repository 主要职责：

- 阵容分页查询；
- 阵容详情查询；
- 新增阵容；
- 更新阵容；
- 删除阵容；
- 批量替换子表成员；
- 公共端读取启用中的阵容列表/详情。

建议方式：

- 创建/更新阵容时，主表保存成功后，删除旧的 `pokemon_lineup_member`，再批量插入新的成员；
- 这种“整包覆盖”的写法与 `starlight_duel` 当前模式一致，易于维护；
- 因为每套阵容最多只有 6 条成员记录，性能不是问题。

### 5.3 Service 设计

Service 层负责所有业务校验：

- 权限校验；
- 精灵数量上限校验；
- 血脉字典合法性校验；
- 性格合法性校验；
- 三项资质字段校验；
- 技能去重校验；
- 保存前统一整理 `sort_order`；
- 审计日志记录。

建议保留一个标准化函数：

```python
def normalize_members(members: list[dict]) -> list[dict]:
    # 去掉空白项、重排顺序、清洗空字符串、校验资质槽位
    ...
```

这样 Ops 与未来批量导入接口可以共享。

    ...

    ...

建议公共列表只返回：

- `id`
- `title`
- `source_type`
- `sort_order`

### 6.3 字典复用接口

阵容系统前端需要这些下拉数据：

- 精灵搜索：复用当前搜索精灵接口模式；
- 技能搜索：复用当前搜索技能接口模式；
- 血脉字典：建议新增公共/ops 字典接口，或直接复用已有 dict 接口；
- 性格字典：已具备公共接口，可直接复用。

## 9. 与现有系统的关系

### 9.1 与 `starlight_duel` 的关系

二者关系建议如下：

- `starlight_duel` 是“专题活动攻略”；
- `pokemon_lineup` 是“通用阵容配置能力”。

也就是说，`starlight_duel` 更像业务专题，`pet_lineup` 更像基础能力模块。

后续有两种演进方式：

#### 方案 A：保持并行

- `starlight_duel` 保留现状；
- 新增 `pokemon_lineup` 独立使用。

优点：

- 风险最低；
- 不影响现有已上线功能。

缺点：

- 两套模块会有一定重复。

#### 方案 B：后续逐步收敛

把 `starlight_duel_episode` 逐步改成：

- 主体是专题信息；
- 精灵配置部分复用 `pokemon_lineup` 数据结构。

优点：

- 长期结构更统一。

缺点：

- 改造成本更高。

**建议当前先采用方案 A**，先把通用阵容系统独立落地，待后续有更多专题攻略时再考虑抽象收敛。

### 9.2 与字典系统的关系

- 性格：走 `personality`
- 血脉：走 `sys_dict`
- 种族值字段：不进数据库字典表，可前后端常量维护

原因：

- 性格与血脉属于业务可运营字典；
- 种族值字段是稳定技术常量，没有必要再建表。

## 12. 关键实现建议

### 12.1 先做“通用阵容”，不要先做“复杂专题”

本需求如果直接和某个玩法强绑定，后面会越来越难复用。建议先把“阵容”定义成基础能力：

- 一套阵容
- 多个精灵成员
- 每个成员一组配置

专题玩法只负责引用这套能力。

### 12.2 三项资质要保存“字段 + 值”

不要只保存 3 个数字，否则前台无法知道这三个值分别对应什么种族项。

正确方式必须是：

- `qual_1_stat_key + qual_1_value`
- `qual_2_stat_key + qual_2_value`
- `qual_3_stat_key + qual_3_value`

### 12.3 允许部分字段留空

攻略类系统的一个关键点是“可表达性”而不是“强约束”。

因此：

- 血脉可空；
- 性格可空；
- 技能可空；
- 单精灵说明可空；
- 阵容说明可空；

这样更符合实际运营使用场景。

---

## 13. 最终结论

基于当前项目，最合适的设计方案是：

- 新增一个独立的通用阵容模块：`pokemon_lineup` + `pokemon_lineup_member`；
- 复用现有 `pokemon`、`skill`、`personality`、`sys_dict`；
- 延续 `starlight_duel` 的“主表 + 最多 6 个成员 + 固定技能槽位”风格；
- 在成员层补充：血脉、性格、三项资质、单精灵说明；
- 在阵容层补充：标题、整体说明、状态、排序、分类。

这样既能满足你现在的精灵阵容配置需求，也能与当前项目的数据结构和后台交互方式保持一致，后续扩展成本最低。
