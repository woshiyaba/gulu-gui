# 精灵详情返回蛋孵化尺寸阈值方案

## 背景

当前 `GET /api/pokemon/{pokemon_name}` 由 `api/routes/pokemon.py` 暴露，实际数据组装在 `api/services/pokemon_service.py#get_pokemon_detail` 中完成。详情响应目前包含基础信息、属性、蛋组、种族值、特性、克制关系和技能列表，但没有返回 `egg_hatch_pet` 中用于蛋实验体型判定的尺寸阈值。

需求是在精灵详情接口中增加当前宠物的蛋孵化尺寸阈值：

- `big_size_length_min`
- `big_size_weight_min`
- `small_size_length_max`
- `small_size_weight_max`

如果当前宠物属于一条进化链，则进化链上所有宠物都复用该进化链第一个宠物的 `egg_hatch_pet` 信息。

## 现状依据

- `pokemon.chain_id` 表示精灵所属进化链。
- `evolution_chain` 使用 `chain_id` 和 `sort_order` 维护同链宠物顺序，`sort_order = 1` 是链上第一个宠物。
- `egg_hatch_pet.pokemon_id` 唯一关联 `pokemon.id`，已有字段单位为：
  - 身长字段：厘米
  - 体重字段：克
- `api/services/pokemon_service.py#get_pokemon_by_body_metrics` 已经在体型匹配逻辑中读取并使用这四个阈值。

## 返回结构建议

在 `PokemonDetailResponse` 中新增独立对象字段：

```json
{
  "egg_hatch_size": {
    "source_pokemon_id": 1,
    "source_pokemon_name": "喵喵",
    "is_chain_reused": true,
    "big_size_length_min": 80,
    "big_size_weight_min": 12000,
    "small_size_length_max": 40,
    "small_size_weight_max": 3000
  }
}
```

字段说明：

| 字段 | 类型 | 说明 |
|------|------|------|
| `egg_hatch_size` | `object \| null` | 蛋孵化尺寸阈值；无 `egg_hatch_pet` 数据时返回 `null` |
| `source_pokemon_id` | `integer` | 实际读取 `egg_hatch_pet` 的精灵 ID |
| `source_pokemon_name` | `string` | 实际读取 `egg_hatch_pet` 的精灵名称 |
| `is_chain_reused` | `boolean` | 当前详情是否复用了进化链第一个宠物的数据 |
| `big_size_length_min` | `integer` | 大块头身长下限，单位 cm |
| `big_size_weight_min` | `integer` | 大块头体重下限，单位 g |
| `small_size_length_max` | `integer` | 小不点身长上限，单位 cm |
| `small_size_weight_max` | `integer` | 小不点体重上限，单位 g |

选择独立对象而不是四个顶层字段，原因是这些字段都来自同一个业务概念，且需要额外说明“数据来源是否为链首宠物”。`null` 也比全 0 更能表达“没有配置蛋孵化数据”。

## 查询规则

### 1. 无进化链

如果当前精灵 `pokemon.chain_id IS NULL`：

1. 直接按当前 `pokemon.id` 查询 `egg_hatch_pet`。
2. 查到则返回当前精灵的尺寸阈值，`is_chain_reused = false`。
3. 查不到则 `egg_hatch_size = null`。

### 2. 有进化链

如果当前精灵 `pokemon.chain_id IS NOT NULL`：

1. 根据 `chain_id` 查询 `evolution_chain` 中 `sort_order` 最小的链首成员。
2. 用链首成员的 `pokemon_name` 回查 `pokemon.id`。
3. 按链首 `pokemon.id` 查询 `egg_hatch_pet`。
4. 查到则所有同链精灵详情都返回链首阈值。
5. 当前精灵 ID 和链首 ID 不同，则 `is_chain_reused = true`；相同则为 `false`。
6. 链首成员没有匹配到 `pokemon` 或没有 `egg_hatch_pet` 时，返回 `null`。

### 3. 进化链数据缺失兜底

如果 `pokemon.chain_id` 有值，但 `evolution_chain` 查不到成员，按当前精灵自己的 `pokemon.id` 查询 `egg_hatch_pet`。这样可以避免脏数据导致详情接口完全拿不到已有尺寸配置。

## SQL 方案草稿

仓储层可以拆成两步，逻辑更直观：

```sql
-- 1. 查询链首宠物名称；chain_id 为空时跳过
SELECT pokemon_name
FROM evolution_chain
WHERE chain_id = %s
ORDER BY sort_order ASC
LIMIT 1;
```

```sql
-- 2. 按链首名称或当前 pokemon_id 查询 egg_hatch_pet
SELECT
    p.id AS source_pokemon_id,
    p.name AS source_pokemon_name,
    e.big_size_length_min,
    e.big_size_weight_min,
    e.small_size_length_max,
    e.small_size_weight_max
FROM pokemon p
JOIN egg_hatch_pet e ON e.pokemon_id = p.id
WHERE p.id = %s;
```

如果想减少数据库往返，也可以用 CTE 实现，但两步查询更容易写清楚 fallback 和调试。

| 当前精灵不存在 | 仍返回原有 `404 {"detail":"精灵不存在"}` |

## 风险与注意事项

- `evolution_chain.pokemon_name` 是基础名，不一定等于具体形态名。链首名称回查 `pokemon` 时，优先精确匹配基础名；如果未来存在多个同名前缀形态，需要明确选择原始形态的规则。
- `egg_hatch_pet` 对 `pokemon_id` 有唯一约束，当前方案默认一只精灵最多一份孵化数据。
- 不建议把没有数据的情况返回全 0，因为表字段默认值本身就是 0，无法区分“配置了 0”和“没有配置记录”。
- 前端如果只需要展示四个阈值，可以直接读取 `egg_hatch_size`；如果需要提示复用来源，可以展示 `source_pokemon_name`。
