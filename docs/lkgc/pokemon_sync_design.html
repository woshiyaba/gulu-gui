洛克观测宠物基础信息同步设计文档

本文档描述通过 fetch_pet_list 分页拉取洛克观测宠物列表， 再按需调用 get_pet_info 获取详情，将本库缺失的宠物基础信息插入 PostgreSQL pokemon 及其关联表的设计。

需求来源：docs/lkgc/feat.txt 第 1 项 列表示例：docs/lkgc/getList_exam.json 详情示例：docs/lkgc/getPetInfo_exam.json 目标表：pokemon / pokemon_attribute 已对比当前 PostgreSQL 数据

目标

  • 分页读取洛克观测宠物列表，逐条构造本库宠物名称并查重。
  • 本库已存在同名宠物时不做任何更新，保证历史数据不被外部数据覆盖。
  • 本库缺失时，上传 main_url 到 OSS，作为 pokemon.image_lc 写入。
  • 读取宠物详情中的 web_pet.catch_way，写入 pokemon.obtain_method。
  • 将洛克观测种族值字段按需求映射到本库 pokemon。

非目标

  • 不更新已存在宠物的基础字段、立绘、种族值、获得方式或头像。
  • 不在本流程同步技能，技能同步由第二项“同步技能”按钮负责。
  • 不自动删除本库中不存在于洛克观测列表的宠物。
  • 不把外部数据作为本库所有字段的唯一权威来源，必填缺省值需要明确规则。

数据库对比结论

以 docs/lkgc/getList_exam.json 的 12 条宠物为样本，已按名称规则对比当前 PostgreSQL 的 pokemon 与 pokemon_attribute。

检查项 结果 设计影响
列表样本数量 共 12 条。 可作为小样本验证，不代表全量覆盖。
按名称命中 10 条命中：迪莫、喵喵、喵呜、魔力猫、火花、焰火、火神、水蓝蓝、波波拉、水灵。 查重应以 pokemon.name 为主，命中后直接跳过，不更新历史数据。
按 source_id 命中 0 条命中。样本的 web_pet_id 与当前库 pokemon.source_id 不是同一套编号。 web_pet_id 只能写入新增记录，不能用来判断已有记录是否相同。
未命中 2 条未命中：鸭吉吉（蓬松的样子）、板板壳（本来的样子）。 这些会进入新增流程，适合验证插入、图片上传和属性映射。
种族值差异 已命中的 10 条，按 sm/wg/mg/wf/mf/sd 映射后与本库一致，差异数为 0。 字段映射方向可保留，但仍不更新已存在记录。
样本属性映射 从命中记录可反推 4 -> 草、6 -> 光、9 -> 火、10 -> 水。 全量实现仍需要完整 type_list -> attribute.id 映射表。
结论：第一项应定位为“按名称补缺”，不是“按外部 source_id 对账更新”。已有数据即使外部编号不同，也不能被覆盖。

数据流程

1. 分页拉列表 调用 fetch_pet_list(page, page_size)，直到返回空列表或达到最大页数。
2. 生成名称 用 name 与 form_name/from 组合为 name（form_name）。
3. 查重 按组合名查询 pokemon.name；存在则跳过，不更新历史记录。
4. 拉详情和图片 调用 get_pet_info(real_pet_id) 获取详情；上传 main_url 到 OSS。
5. 插入本库 写入 pokemon，再写 pokemon_attribute 等关联表。

本库必填字段处理

  • pokemon.trait_id 是必填外键。建议预置一个“无/未知”特性，导入时作为默认值。
  • pokemon.type、type_name、form、form_name 可按现有字典规则写默认阶段/形态；至少保证空字符串不破坏前端展示。
  • pokemon.egg_group 和 pokemon_egg_group 可先留空，后续再根据 egg_type_list 设计映射。
  • 属性映射缺失时，建议跳过该属性并记录告警；如果最终没有任何属性，可允许宠物先入库但在结果里标记待补属性。

本次样本只能覆盖 4 个洛克观测属性 ID，完整同步前仍需补全 18 个属性映射。

未覆盖的 type_list ID 不能靠猜测写库。实现时应使用显式映射表； 映射缺失时记录告警，并允许该宠物先入库或根据产品要求直接跳过。

幂等与事务

  • 单只宠物以 pokemon.name 查重，存在即跳过，保证重复运行不会覆盖人工维护数据。
  • 每只宠物独立事务：图片上传成功后再写库；数据库写入失败则回滚该宠物。
  • 同步过程输出统计：扫描数、已存在跳过数、新增数、图片失败数、详情失败数、属性映射缺失数。
  • 失败明细保留宠物名称、real_pet_id、失败阶段和错误消息，便于补跑。

推荐接口形态

建议放在运营后台接口下，避免作为公开接口暴露：

POST /api/ops/pokemon/sync-lkgc

Request:
{
  "page_size": 100,
  "max_pages": 120,
  "dry_run": true
}

Response:
{
  "scanned": 1200,
  "inserted": 83,
  "skipped_existing": 1110,
  "failed": 7,
  "warnings": ["属性映射缺失: type_id=999"],
  "items": [
    {
      "name": "石肤蜥（本来的样子）",
      "real_pet_id": "127-1",
      "status": "inserted",
      "message": "已新增"
    }
  ]
}

推荐写入顺序

  1. 分页拉取列表，规范化响应为宠物数组。
  2. 根据 name 和 form_name/from 生成目标名称，并查 pokemon.name。
  3. 命中则记录 skipped_existing，不再调用详情、不上传图片。
  4. 未命中则调用 get_pet_info(real_pet_id) 获取 catch_way 等详情字段。
  5. 上传 main_url 到 OSS，得到 image_lc。
  6. 按字段映射组装 pokemon payload，插入主表。
  7. 按 type_list 映射写入 pokemon_attribute。
  8. 返回同步统计和逐条处理结果；如果是 dry_run，数据库事务回滚。

待确认项

1. 属性映射表。 当前样本仅反推出 4->草、6->光、9->火、10->水，仍需确认 lkgc type_list 到本库 attribute.id 的完整映射。

2. 默认特性。 需要确定用于自动导入的默认 pokemon_trait，因为 pokemon.trait_id 不能为空。

3. 编号格式。 需要确认新增宠物的 pokemon.no 使用 real_pet_id、pet_id 还是转换为现有 NO.xxx 风格。

4. 前端入口。 可做成精灵维护页的批量同步按钮，也可先作为后端接口或脚本执行，待确认交互范围。
