洛克广场同步精灵数据 · 设计文档

同步「洛克广场」(lkgc) 云函数 m-pet-co 提供的全量精灵数据，仅插入未录入精灵，保留本库已维护的属性、特性、技能与种族值历史；本次设计已对照本库现存 460 条 pokemon 做了差异校准。

模块: api/ops · OpsPokemonView 云函数: getList / getPetInfo 数据源域: env-00jxhb62nv6n 对应 feat.txt 第 1 部分

1. 背景与目标

当前本库的 pokemon 表来源于多个历史采集脚本，需要补齐「洛克广场」小程序最新的精灵列表，主要目的：

  • 新增本库尚未录入的精灵（命中即跳过，永远不覆盖历史数据）。
  • 把不稳定的微信 CDN 主图 (mmbiz.qpic.cn) 落到自家 OSS，保证前端图像可用性。
  • 统一种族值口径（mf/mg/sd/sm/wf/wg → mdef/matk/spd/hp/def_val/atk）。
  • 补全 obtain_method，并在有头像 (avatar_url) 时回填 pokemon.avatar。
本期只处理「精灵基础信息」同步；技能同步已在精灵维护列表行操作内复用POST /api/ops/pokemon/{id}/sync-lkgc-skills（feat.txt 第 2 部分）。

2.2 getPetInfo 关键补充

列表只有种族值与立绘 URL，捕获方式必须通过详情拿到：

  • web_pet.catch_way：标准捕获描述，例 "在轻风山，战斗概率出现。" → 映射到 pokemon.obtain_method。

3.1 命名习惯差异（关键陷阱）

陷阱：lkgc 与本库的 form 填法不完全一致。
  • 本库：历史上很多单形态精灵也填 form_name="本来的样子"（如「古钟蛇（本来的样子）」「鸭吉吉（蓬松的样子）」）。
  • lkgc：单形态精灵会留空 form_name="" 或 null，仅多形态时才填具体名字。
所以 直接用 name（form_name） 严格匹配会漏命中，本次抓取里就出现了 2 条这种冲突（参见 §3.2.3）。

6.0 拉取后过滤（sanitize，强制）

目标：业务代码、日志、响应、审计里只流转「规范列表项」，不把 lkgc 内部坐标暴露出去。

  1. 解析 HTTP：得到原始 dict 列表 raw_items（可含 real_pet_id、Mongo _id 等）。
  2. 拆出详情 RPC 入参：对每条原始 dict，在生成规范项之前执行 detail_arg = raw.pop("real_pet_id", None)（无则 None）。将非空的 detail_arg 记入侧表 detail_arg_by_base_id[raw["base_id"]] = detail_arg，该表仅存在于本次同步函数调用栈内，供 S3 分支调用 get_pet_info(pet_id=detail_arg)；RPC 返回后不得写入任何持久结构。
  3. 构造规范项：在已 pop 的前提下，再 sanitized = {k: v for k, v in raw.items() if k not in DROP_KEYS}，其中 DROP_KEYS 至少包含 _id（及任何其它仅 lkgc 内部使用的键）；后续 S1/S2/S3 分桶、_match_key、OSS、入库只读 sanitized。
  4. 合并详情：详情 JSON 同样过一遍 DROP_KEYS 后再与 sanitized 合并；禁止把详情里可能出现的同源字段写回规范项（若存在则继续剔除）。
  5. 调试过滤：OPS 请求体使用 base_id_filter（见 §7.1 / §8.4），禁止再提供按 real_pet_id 过滤的入口。
实现位置建议：在 api/services/ops_service.py 的同步入口最先调用 _sanitize_lkgc_list_row；或在 common/utils/lkgc.py 增加「解析后立刻 strip」的包装函数，由 Service 统一调用，避免各调用点漏删。

6.1 三段命中策略（根据 §3 实测调整）

实测发现 lkgc 与本库的 form_name 填法不一致（lkgc 单形态留空、本库历史写"本来的样子"），直接的严格匹配会把同一只精灵当成新增。改为三段决策：

步骤判定条件动作预期数量（本次抓取）
S1 严格命中 name（form_name） 全等本库某行 命中跳过，整条不写 344
S2 同基名冲突 lkgc form_name 为空 / null，但本库存在 {name}（%） 的同基名行 视为命中，不新增；记录到 conflicts，列出本库已有的 form 列表，留待人工核对 2（古钟蛇、寒音蛇）
S3 真新增 S1、S2 都不命中 进入新增分支：拉详情 → 上传 OSS → 写库 30
S2 之所以「视为命中而非新增」，是因为如果直接按裸名插入，会和已有 古钟蛇（本来的样子） 形成两条「指向同一精灵」的记录，破坏前端按 name 唯一展示的隐含约定。冲突条目在响应里独立分桶，方便管理员人工确认。

6.3 与本库历史数据的协同

  • 命中分支完全跳过：S1/S2 不写任何字段，保护本库已维护的 obtain_method、trait_id、avatar、source_id 等所有内容（注意线上数据显示 obtain_method 多为「主线剧情（人手一只）」这类人工编辑文本，lkgc 的 catch_way 不能盲目覆盖）。
  • S3 新精灵的 no：取 DB 中 NO.XXX 数字部分最大值（当前 NO.347），按列表顺序连续递增分配；本次预期写入 NO.348 ~ NO.377。
  • S3 新精灵的 source_id：写入 lkgc 的 base_id（官方游戏图鉴 id，100% 覆盖且唯一）；不使用 real_pet_id（lkgc 内部坐标系）和 web_pet_id（127/376 为 -1）。
  • avatar：本期 lkgc 数据中只有 12 条非空，S3 新精灵优先用 lkgc avatar_url，否则字段写空。
  • image_lc：S3 新精灵必填，落 OSS 后写入。

6.4 幂等性

三段策略对相同 lkgc 输入再次执行只会重复跳过 / 重复决策为「真新增」（如果上一轮失败回滚），整体幂等。建议同步前用 pg_advisory_lock(hashtext('lkgc_sync')) 防止并发触发。

6.5 OSS 上传规则

  • 主图 main_url：oss.oss.COSClient.upload_from_url(url, prefix="pokemon/main")。
  • 头像 avatar_url：非空时 upload_from_url(url, prefix="pokemon/avatar")，否则字段写空。
  • 下载或上传失败：当前条记录到 warnings，image_lc / avatar 写空，仍尝试入库其余字段。
  • 注意 lkgc main_url 部分来自 env-00jxhb62nv6n.normal.cloudstatic.cn，URL 含 expire_at 签名，必须同步立即上传，不能延迟。

6.6 trait_id 缺失处理（NOT NULL 约束的核心难点）

本库 pokemon.trait_id NOT NULL；lkgc 列表只有特性 id，详情才有名字。同步流程（规范项已无 real_pet_id，详情 RPC 见 §6.0）：

  1. 对 S3 新精灵调用 getPetInfo，从 feature_list[*].name 取第一个非空特性名。
  2. 按 pokemon_trait.name 查找（精确匹配）。
  3. 命中：写入 trait_id。
  4. 未命中：
    • 生成告警 缺失特性: 刺肤(feature_id=37)，跳过该精灵插入。
    • 响应中将该条标记 action="skip_missing_trait"。
    • 运维先在「字典/特性」页补齐 pokemon_trait 后再次触发同步即可。
特性属于核心维表，自动新增风险高（desc、icon、effect 全靠人工校对），故采取「告警+跳过」。最终新增数 = 命中特性的未匹配精灵数。

6.7 no / source_id 编号策略

线上 GET /api/pokemon 实测本库 no 100% 是 NO.XXX 三位数字格式，最大 NO.347，不使用 real_pet_id 作 no 来源（lkgc 内部坐标系，不应暴露）。本期为 S3 新精灵生成：

6.7.1 no

# Service 启动时一次性算出基准号
SELECT MAX(CAST(SUBSTRING(no FROM 'NO\.(\d+)') AS int))
FROM pokemon
WHERE no ~ '^NO\.\d+$'
=> 347

# 然后按列表顺序连续分配
NO.348, NO.349, ..., NO.377  (S3 共 30 条预期)

说明：

  • 30 条真新增都是 30 个互不相同的基名（无同基名多形态），所以一只精灵分一个 no，不存在 form 共享 no 的复杂场景。
  • pokemon.no 列没有唯一约束（参考 sql/wikiroco.sql），但本期算法已保证全局递增、不冲突。
  • 若中途有人并发新增精灵，建议同步前先 pg_advisory_lock，避免 MAX(no) 期间被插入。

6.7.2 source_id

  • 写入 lkgc 的 base_id（int，范围 3001~3745，100% 覆盖且全唯一）。
  • 不使用 real_pet_id（lkgc 内部坐标系；用户明确禁用）。
  • 不使用 web_pet_id（376 条中 127 条为 -1，覆盖不足）。
  • base_id 看起来就是官方游戏的图鉴 id，相对 lkgc 自家的 pet_id 等字段更稳定，即便 lkgc 后续换字段命名也能基于此重新对齐。

6.7.3 与 §6.0 一致：禁止 real_pet_id 进入数据载体

  • real_pet_id 不得出现在规范列表项、合并后的写入 payload、OPS 响应 items[*]、审计 JSON、前端状态树中。
  • 若云函数仍要求用该值调 getPetInfo，只能按 §6.0 在解析原始响应的同一调用栈内 pop 出临时字符串，作为 RPC 实参；RPC 结束后不得再持有。
  • 若 lkgc 侧不再接受该入参或字段为空，详情拉取失败则走 §6.8 退化分支。
  • 对外调试与按条重跑一律用 base_id（base_id_filter）。

skills (基础 / 血脉 / 学习)详情 skill_list该精灵跳过技能写入；后续可在「同步技能」入口里单独刷
image_lc / avatar / 6 项种族值 / 类型详情可覆盖 getListgetList 即可，不受影响

结论：即便详情完全失效，S3 新增对于本库已有的 trait 名称恰好命中的精灵仍能成立（约束：pokemon.trait_id NOT NULL）；否则按缺特性跳过。

备选：如果想在「无详情」时也能解析 trait，可以单独建一份 lkgc feature_id (int) → pokemon_trait.id 的本地字典，落到 docs/lkgc/trait_id_map.json，启动时加载。本期不强制构建，先依赖详情拉取。

7.3 Service 核心伪代码（三段分桶）

async def sync_pokemon_from_lkgc_for_ops(user, options):
    ensure_role(user, {"admin"})                # 仅 admin 可触发整体导入
    raw_items = await asyncio.to_thread(_fetch_all_pets_lkgc_raw, options)

    # 0. 拉取后立刻过滤（§6.0）：每条拆成 (sanitized, detail_rpc_arg)；后续只传递 sanitized dict
    rows = [_split_and_sanitize_lkgc_row(dict(r)) for r in raw_items]
    items = [san for san, _arg in rows]
    detail_arg_by_base_id = {san["base_id"]: arg for san, arg in rows if arg}

    # 1. 计算严格 key 与基名（只用规范项）
    strict_keys = [_match_key(it) for it in items]                 # name（form_name）
    base_names  = [_clean(it["name"]) for it in items]

    # 2. 一次性查 DB
    existing_strict = await ops_repository.fetch_existing_pokemon_names(strict_keys)
    db_base_index   = await ops_repository.fetch_pokemon_names_by_base(set(base_names))
    # db_base_index: { "古钟蛇": ["古钟蛇（本来的样子）", "古钟蛇（本命年的样子）"], ... }

    # 3. 三段分桶
    matched, conflicts, todo = [], [], []
    for it, k, base in zip(items, strict_keys, base_names):
        if k in existing_strict:
            matched.append(it)                                     # S1
        elif not _has_form(it) and db_base_index.get(base):
            conflicts.append((it, db_base_index[base]))            # S2
        else:
            todo.append(it)                                        # S3

    # 4. 仅 S3 才拉详情（节省 ~90% 请求）；pet_id 实参从 detail_arg_by_base_id 取，RPC 后即弃
    details = await _bounded_gather(todo, lambda san: get_pet_info(pet_id=detail_arg_by_base_id[san["base_id"]]), concurrency=5)

    trait_names = {_pick_trait_name(d) for d in details if d}
    trait_map = await ops_repository.fetch_traits_by_names(trait_names)

    # 5. 逐条上传 OSS + 入库
    results = [{"action": "matched", ...} for it in matched]
    results += [{"action": "conflict_form", "conflict_with": forms, ...} for it, forms in conflicts]
    for san, detail in zip(todo, details):
        try:
            payload = _build_pokemon_payload(san, detail, trait_map, options.dry_run)
            if not payload["trait_id"]:
                results.append({"action": "skip_missing_trait", ...})
                continue
            pid = await ops_repository.insert_pokemon_from_lkgc(payload)
            results.append({"action": "inserted", "pokemon_id": pid, ...})
        except Exception as exc:
            results.append({"action": "skip_failed", "message": str(exc)})

    await ops_repository.create_audit_log(
        user_id=user["id"], resource_type="pokemon",
        action="sync_from_lkgc", after_json=summary,
    )
    return summary

                   RETURNING id""",
                (...payload values...),

7.5 并发与限流

  • 同步全量 376 条 ≈ 4 次 getList + 仅 ~30 次 getPetInfo（S3 真新增分支才需要详情）。
  • 详情请求建议并发 5 + 每条间隔 200~500ms，避免触发云函数频控。
  • OSS 上传走 asyncio.to_thread，最大 3 并发即可（30 张图，总耗时约 15~30s）。
  • 预估完成时间 30~60 秒（比初版 60~180s 大幅缩短，因为 S1+S2=346 条不再需要详情）；接口同步等待。

7.6 权限与审计

  • 角色限制：admin（高风险批量写库）。
  • 审计：ops_audit_log.action = "sync_from_lkgc"，after_json 写入 summary（不含逐条 items，避免日志膨胀）。
  • 逐条 items 仅返回给前端展示，不入审计。

9. 边界与待确认

#问题建议处理
Q1本库 attribute.id 是否与 lkgc type_list 完全对齐（1~18）？需要 admin 在执行前比对一次 SELECT id, name FROM attribute ORDER BY id，确认无错位；若不对齐需要建立 type_id → attribute.id 字典映射
Q2本库 pokemon_trait.name 与 lkgc 详情 feature_list[*].name 是否完全同名？对差异条目走「告警 + 跳过」，不做自动新增（避免污染特性主表）
Q3S2 冲突案例（古钟蛇 / 寒音蛇）本期处理方式？本期一律「视为命中」跳过，弹窗仅展示提示，不主动写入
Q4pokemon.no 用什么生成？采用 MAX(no_num) + N 递增，沿用本库 NO.XXX 三位数风格（本期分配 NO.348 ~ NO.377）。不使用 real_pet_id（lkgc 内部坐标系）
Q4-补同步过程中如果有人并发新增精灵，会不会 no 冲突？推荐 SELECT pg_advisory_xact_lock('pokemon_no_seq'::regclass::oid::bigint) 串行化；冲突概率本身极低
Q4-外pokemon.source_id 取哪个 id？取 lkgc base_id（3001~3745，100% 覆盖且唯一）；不取 web_pet_id（127 条为 -1）和 real_pet_id（用户禁用）
Q5是否引入 chain_id / 进化链？本期不引入；lkgc 的 pre_ids 可作后续单独任务（与 base_id / 名称族谱对齐，不依赖已剔除字段）
Q6命中行为是否包含「主图 / 头像 OSS 重传」？不包含；命中即完全跳过该精灵的所有写入。后续若要单独维护图，建议做一个独立的「图床刷新」工具
Q7同步是否要做 pg_advisory_lock 防并发？推荐做；如运维不需要，可在配置里关掉
Q8是否需要异步任务化（防 HTTP 超时）？修正后预估 30~60s，FastAPI 默认无 60s 限制；先做同步；后续 PR 可迁到队列
